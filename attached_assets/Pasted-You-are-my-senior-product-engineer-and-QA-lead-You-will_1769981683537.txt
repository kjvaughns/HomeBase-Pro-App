You are my senior product engineer and QA lead. You will implement, wire, and verify the full HomeBase mobile app experience so that the Service Pro side is the primary experience, while still supporting Homeowner accounts. Build everything end to end with correct data models, permissions, Stripe Connect flows, and working UI.

Primary goals
1. The app must open to a role selection screen, not a homeowner focused home screen.
2. Users can be both Homeowner and Service Pro under one login. They can switch roles any time.
3. The Service Pro workflow must fully replace pen and paper: client CRM, scheduling, invoices, payments, booking link intake, and automated confirmations.
4. All core flows must be functional and testable with clear acceptance criteria.

Platform assumptions
Mobile app with authentication, database, and Stripe Connect already integrated or available.

Account model and permissions
User can have two profiles tied to the same auth user_id:
1. homeowner_profile
2. provider_profile

Rules
1. Same login can create both profiles.
2. Switching roles changes the navigation and default home screen.
3. Data access is role scoped:
   Homeowner sees their own bookings, invoices, payments, messages, saved providers.
   Provider sees their business, clients, schedule, invoices, payouts, booking links.

AI intake and quoting behavior
When a homeowner uses booking link
1. They describe their issue in free text.
2. The system asks a small number of dynamic follow up questions to clarify.
3. The intake submission is created and visible on the provider side.
4. The quote shown to homeowner can be either:
   Range quote (low, high)
   Fixed quote
   Or “Estimate after review” depending on provider settings.
Implement this in a deterministic, reliable way. If an AI model is used, include a fallback question set per category so it always works.

Acceptance criteria and QA checklist
Role gateway
1. App never lands on homeowner home by default.
2. Logged in users always see role selection unless they selected “remember my choice”.
3. Users can add the second role later without creating a second login.
4. Switching roles changes tabs and dashboard correctly.

Service Pro core workflows must work end to end
1. Create client
2. Create job on schedule
3. Create invoice linked to client and job
4. Send invoice link
5. Pay invoice using Stripe Connect routing
6. Invoice becomes paid automatically
7. Payment record created
8. Receipt sent and visible
9. Funds screen reflects balances and payouts

Booking link flow must work end to end
1. Provider creates booking link
2. Homeowner opens link, submits intake, sees quote or next steps, books time, pays deposit if required
3. Job appears on provider schedule with intake details
4. Provider can convert intake to job and invoice
5. Notifications are sent

Data integrity
1. Provider cannot see other providers’ data
2. Homeowner cannot see other homeowners’ invoices or jobs
3. All records are tied to provider_id and user_id properly

Performance and reliability
1. No screen loads can be blocked by slow network calls
2. All loading states and empty states are present
3. Error messages are clear and actionable

Deliverables
1. Implemented screens and navigation
2. Implemented database tables and policies
3. Implemented Stripe Connect onboarding, payment routing, and webhooks
4. Implemented booking link and intake workflow with AI plus fallback
5. QA pass with the checklist above, plus a short written test report describing each tested flow and proof that it works

Do not add extra features beyond this scope. Focus on making these flows extremely stable and simple.
